Обучение Git

  • Git – популярная система для хранения программного кода, (система контроля версий)

    Запомнить! Система контроля версий (CVS - Control Version System) - позволняет вернуться к определённой версии файла или определить, кем и когда были сделаны изменения в файле.

    Запомнить! Репозиторий Git – хранилище для истории разработки проекта, которое содержит все версии используемых файлов.

    Запомнить! Коммит (commit) - фиксация состояния проекта в репозитории Git

    Запомнить! Индекс – список файлов, изменения в которых система Git отслеживает. В индексе также предварительно сохраняются все изменённые файлы, которые будут зафиксированы в репозитории при следующем коммите

    Запомнить! Ветка – изолированный независимый поток разработки, в котором можно делать коммиты, недоступные для других веток

    Основные возможности git:

    1. Добавлять новый функционал.
    2. Исправлять ошибки.
    3. Работать в команде.

  • Первым делом нужно указать своё имя и адрес электронной почты.

    Эту информацию Git будет включать во все создаваемые коммиты и именно по этим реквизитам устанавливается авторство изменений в общем репозитории.

    Имя и адрес электронной почты удобнее сохранить в глобальные переменные с помощью следующих команд:

    git config --global user.name "Sergey Stryuchkov"

    git config --global user.email "serega.stryuchkov@gmail.com"

    git config --list - все установленные параметры Git

    git init - инициализирует систему Git для текущей папки и создаст в ней новый пустой репозиторий.

    В результате в текущей папке будет создан каталог .git, в котором будут находиться все необходимые для git файлы.

  • git status - статус файлов которые были добавлены, изменены, удалены...

    git add (имя файла) - добавляет файл в индекс, что-бы git следил за ним.

    git add . - добавляет все файлы внутри текущей директории в индекс, что-бы git следил за ним.

    • *** Файлы, находящиеся в индексе Git и готовые к включению в коммит.
    • *** *** Файлы, за которыми Git следит, но информация об их изменениях не будет добавлена в коммит, так как последние версии этих файлов еще не добавлены в индекс.
    • *** *** *** Файлы, за изменениями которых Git не следит.

  • git commit - создаёт коммит. => откроется окно и нужно придумать ему имя.

    git commit -m 'name' - создаёт коммит. При этом имя указывается сразу.

    git log - История коммитов.

    1. commit 3dd34dfd80b... - хеш коммита (уникальный идентификатор коммита)
    2. (HEAD -> master) - в какой ветке был совершён коммит
    3. Author - кем был совершён коммит
    4. Date - когда был совершён коммит
    5. Первый тестовый коммит: - сообщение коммита

    git log --oneline - git log в компактном варианте.

    git diff 'id' - показывает изменения в коммите.

    git diff - показывает изменения в текущем коммите.

  • git-проект состоит из:

    Рабочий каталог - текущие файлы над которыми введётся работа. Если создать или изменить файл, он сохранится в рабочем каталоге. Рабочий каталог не имеет отношения к папке .git.

    Индекс - находится в папке .git и содержит информацию о том, какие файлы и в каком состоянии будут зафиксированы при следующем коммите.

    HEAD - указатель, который обычно указывает на последний комит в ветке.

    Обработка файлов в Git происходит следующим образом:

    1. Мы редактируем файл в рабочем каталоге. У такого файла будет состояние "Изменен" (modified).
    2. Индексируем изменённый файл, добавляя его в индекс командой git add. Статус такого файла – "Индексирован" (staged)
    3. Сохраняем изменения в репозитории, выполняя коммит с помощью git commit. Статус такого файла – "Зафиксирован" (fixed).

    Отменение изменений

    git reset --soft/--hard - Откатиться к нужному коммиту.

    git reset --soft 'id' - переместит указатель HEAD на нужный коммит, при этом изменения которые произошли после коммита к которому хотим перейти останутся.

    git reset --hard 'id' - удаляет файлы и полностью возращает состояние проекта к текущему коммиту к которому мы хотим откатится).

    Запомнить! Не обязательно копировать полностью идентификатор (будет достаточно 7 символов или больше)

    git reset --soft

    1. Откатимся к нужному коммиту. Теперь HEAD переместился на этот коммит.
    2. Произведём изменения.
    3. Добавим все файлы в индекс и создадим коммит. (git add . => git commit)
    Можно откатить файл или группу файлов к коммиту которому перешли.
    1. git reset --soft
    2. git reset HEAD 'название файла'. Теперь этот файл имеет статус unstage (git за ним не отслеживает)
    3. git checkout -- 'имя файла'. Полностью откатывает измненеия до текущего коммита.
    4. Добавим все файлы в индекс и создадим коммит. (git add . => git commit)

    git reset --hard

    1. Откатимся к нужному коммиту.
      Теперь изменения которые состоялись после коммита к которому откатились - удалены и восстановить их нельзя.

    Запомнить! git reset --hard использовать в очень крайних случаях.

  • git branch - Проверить какие ветки есть в проекте.

    Новый функционал разрабатывают в новых ветках, изолированно от основной ветки проекта. Когда все нужные изменения сделаны, функционал полностью готов и протестирован, ветка объединяется (сливается) с основной.

    Имя для ветки нужно выбирать осмысленно, чтобы было понятно ее назначение (реализация нового функционала или исправление выявленных ошибок). Например: feature-cart (корзина)

    Все ветки существуют параллельно друг другу, разработка в них ведётся независимо

    Создание и переключение веток

    git checkout -b 'имя_ветки' - Создаёт новую ветку и сразу переходит в неё.

    git checkout 'имя_ветки' - Переключение между ветками.

    Слияние веток

    Например вся работа в новой ветке выполнена. Теперь эту ветку нужно слить с основной.

    Для этого нужно переключится в ветку, в которую надо добавить изменения (master).

    git merge 'имя_ветки' - Слияние веток.

    Удаление веток

    После того, как произошло слияние двух веток, вторая ветка больше не нужна.

    git branch -d 'имя_ветки' - Удаление ветки (d - delete)

  • Если изменения в ветках затрагивали разные строки одного файла, то скорее всего Git сможет слить эти файлы без ошибок.

    Но если изменения в разных ветках затрагивают одни и те же строки в файле – в этом случае возникнет конфликт слияния.

    1. Создадим новую ветку bug-cart (git checkout -b 'bug-cart)
    2. Допустим мы изменили файл cart.html сразу в двух ветках, и добавили эти изменения в коммит в разных ветках (git add . => git commit -m)
    3. Перейдём в основную ветку (git checkout master)
    4. Объединим ветки с помощью git merge bug-cart
    5. Получим ошибку, в git status смотрим где произошла ошибка.
      В git diff можно посмотреть подробно о ошибке.

    7. И выбираем что нужно оставить, в данном случае нужно оставить то, что было добавлено в ветке bug-cart:

    8. Создаём новый коммит (git add. => git commit -m)
    9. Теперь если нужно, можем удалить ветку bug-cart (git branch -d bug-cart)

  • Иногда возникает необходимость оперативно переключиться в другую ветку, но в текущей рабочей ветке файлы еще не готовы для коммита. В этом случае Git позволяет сохранить сделанные изменения, не создавая при этом коммит.

    git stash - Собирает незакоммиченные изменения, удаляет их из файлов и в специальном виде архивирует их в Git (помещает в стек незавершенных изменений).

    git stash list - Посмотреть что находится в stash

    git stash apply - Добавит последний добавленный stash

    git stash apply stash@{0} - Добавит конкретный stash

    git stash pop - Добавит последний stash и сразу удалит из списка stash

    git stash drop stash@{0} - Удалит запись из stash

    Например, в ветке bug-cart что-то разрабатывается, и срочно нужно что-то поменять в другой ветке:

    1. Выполняем команду git stash
    2. Переключаемся к нужной ветке, делам работу которую требуют. (добавляем в индекс, в коммит...)
    3. Возращаемся назад к ветке где архивировали файлы (git stash)
    4. Добавляем stash
    5. Удаляем этот stash из списка

  • GitHub

    1. Создание нового репозитория на GitHub
    2. Загрузка локального репозитория на GitHub

      Чтобы загрузить локальный репозиторий на GitHub, нужно сообщить Git'у адрес нужного нам удалённого репозитория

      git remote add origin ...{link}

      origin - имя (может быть любым) У одного проекта может быть несколько удалённых репозиториев под разными именами (разные стадии проекта могут заливаться на разные ресурсы).

    3. git push

      git push удалённый_репозиторий ветка - для выгрузки содержимого локального репозитория в удаленный репозиторий. Она позволяет передать коммиты из локального репозитория в удаленный.

      Ключ -u (upstream) здесь указывает на то, что мы заливаем на сервер ту ветку (master), которой там ещё нет. Во время выполнения команды GitHub запрашивал учетные данные (имя и пароль) для доступа к репозиторию.

      git push -u origin master

    Обязательные файлы на GitHub

    1. .gitignore содержит список файлов и каталогов, за которыми Git не должен следить и которые не будут сохраняться в репозитории.
    2. readme.md - описанием проекта.
    Запомнить! Пустые папки Git игнорирует.

    Клонирование репозитория с GitHub на локальный компьютер

    1. В GitHub копируем ссылку репозитория (code --> HTTPS)
